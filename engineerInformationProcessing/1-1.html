소프트웨어 공학 (SE : Software Engineering)

소프트웨어 공학은 소프트웨어의 위기를 극복하기위한 방안으로 연구된 학문이다. 소프트웨어의 위기라 하면. 개발한 소프트웨어를 사용함에 있어서 있을 수 있는 여러가지 문제를 말하는거 같은디… 여러가지 방법, 도구, 관리기법 을 써서 품질과 생산성 향상을 도모한다.

소프트웨어 공학의 3원칙

  1. 현대적 프로그래밍 기술을 계속적으로 적용한다.
  2. 지속적으로 검증한다.
  3. 개발 관련 사항 및 결과를 명확히 기록하여 유지한다.

소프트웨어 생명 주기 (Software Life Cycle)

소프트웨어 생명주기 영어로 허면 소프트웨어 라이프사이클 Software Life Cycle. 셤에서 영어로 물어볼 수도 있응게 영어도 외워둬유. 근디 영어로 적으라 하지 않았으면 굳이 영어로 적을 필요는 없어야. 영어 틀리면 오답처리유. 소프트웨어 생명주기란 소프트웨어를 개발하기 위한 설계,운용,유지보수 등의 과정을 각 단계별로 나눈것 나선형 모형, 폭포수 모형, 프로토타입 모형, 애자일 모형 등.

나선형 모델 (Spiral Model)

나선따라 뱅글뱅글 돌아가믄서 점진적으로다가 소프트웨어를 거시기혀서 완백한 최종 소프트윈어를 맹드는 것인디... 보햄(Boehm)이 제안함.

4가지 주요활동

  1. 계획수립
  2. 위험분석
  3. 개발 및 검증
  4. 고객평가

4번에서 고객이 완성된 소프트웨어를 사용해보고 평가를 하는디 맴에 차지 않으면 수정해달라고 빠꾸를 먹일것이유. 그람 다시 1번으로 돌아가는거유.그렇게 ㅈ나게 뺑뺑이 돌것이유. 뺑뺑이 돌다부면 시간이 부족혀서 야근도 하고 철야도 하고…. 결국 고객이 이만하면 완벽하제. 통과! 사인을 줄 때가 올거유. 그때 비로소 개발이 끝나는 거시 바로 나선현모형이유. 셤에는 4가지 주요활동 순서를 적으라고 할 것잉께 순서 단디 알아둬유.

폭포수 모델 (Waterfall Model)

가장 오래된 생명주기 모델. 폭포수가 한번 떨어져뿔면. 다시 올라가지 못한는 맨치로. 폭포수 모형에서 한번 단계를 넘거가뿔면 이전 단계로 올라갈 수가 없슈. 그니께 어떻혀야것수. 단계 넘어가기 전에 철저하게 검토하고. 승인과정을 거쳐서. 아 이거 문제 없구마. 할때 다음단계로 넘어가는 것이유. 워매 숭한거. 지가 신입때 짠 코드를 자꾸 빠구먹이는 박부장이 생각났는디. 그때 징허게 힘들었슈.

프로토타입 모형 (Prototype Model)

고객의 요구사항을 알기위해서 견본품을 미리 맨들어 보는 것이유. 견본품을 맹든다 허면 프로토타입 모형이라 알아두면 되는것이유.

애자일 모형 (Agile Model)

애자일 모형은 말이유. 고객이 이거 해랐다가. 아닌가벼. 미안혀. 저렇게 개발 부탁혀. 막 이러는데 워매 환장하겠는거. 이럴때도 잽싸게 대응가능하도록 하는 모델이유. 빠르고 낭비없는 고객과의 소통이 애자일 모형의 키포인트인 것이유. 스크럼(Scrum) , 익스트림 프로그래밍 (XP, eXtreme Programming), 칸반(Kanban), Lean, 기능중심개발 (FDD:Features Driven Development)

에자일의 핵심가치

  • 프로세스나 도구보다는 개인과 상호작용
  • 문서보다는 실행되는 결과물(소프트웨어)
  • 협상보다는 협업
  • 계획보다는 변화에 반응

스크럼 (Scrum)

애자일 방법론에서 스크럼(Scrum)이 있었슈. 스크럼은말이유. 어떤 현장에서 모두가 어깨를 마주대고 서로를 단단히 결속하는 모양새를 스크럼을 짠다고 하는것이유. 이렇게 여럿이서 하나의 팀을 구성해서. 팀을 중심으로 개발 효율을 높이는 방법이유.

스크럼의 구성원

  1. 제품책임자 (PO : Product Owner)
    • 요구사항이 담긴 Backlog를 작성
    • 제품에 대한 이해도 높고. 요구사항 책임지고. 의사 결정할 사람.
  2. 스크럼마스터 (SM : Scrum Master)
    • 스크럼 팀이 잘 수행할 수 있게 가이드역할.
  3. 개발팀 (DT : Development Team)
    • 스크럼 팀원

셤에서는 백로그 작성하는 주체가 누구냐고 물어볼 수 있어유. 제품 책임자 라고 답하면 돼유.

스크럼 개발 순서

  1. 스트린트 계획 회의
  2. 스프린트
  3. 일일 스크럼 회의
  4. 스프린트 검토 회의
  5. 스프린트 회고

익스트림 프로그래밍(XP : eXtreme Programming)

고객이 자꾸 이랬다 저랬다 허도. 거기에 유연허게 대응할려고. 아싸리 고객을 개발과정에 참가시켜버리는 방법이유. 긍께… 사무실에 고객도 같이 출근혀서.. 개발자가 고객한테 완성된 소프트웨어를 수시로 보여주는거유. 고객님 이거좀 확인해봐유. 잘됐쥬? 아니 잘되기는 뭐가 잘돼야. 앱이 죽잖여. 웜매? 어쩌다 죽는겨? 봐바유. 아따. 고객님 폰에서 문제가 있구마. 문제 확인헜소. 바로 싸게 고칠랑게 기다리라고. 막 이러는 것을 반복하면서 생산성을 향상시키는 방법이란 말이유. 그럴려면 고객과 의사소통(Communication) 이 잘 되야 할 것이고 의사소통은 오해 없도록 단순해야(단순성 Simplicity) 할 것이고. 고객한테 문제가 있을 지도 모르는 소프트웨어를 바로 보여줘야 항깨 용기(Corage)도 있어야 할 것이고. 아니 앱이 뭐 이따구여. 개발 할겨? 말겨? 시방 뭐라고 혔어? 막 그래 욕허고 그래뿔면 개발 진행이 안되고 싸움만 날것잉께. 상대를 존중(Respect) 하는 자세도 필요할 것이고. 뭐 찔러보면 응답이 와야하는게 인지상정 아니겠슈.피드백(Feedback) 역시 중요한 것이유. 개발자가 개발 완료된 소프트웨어를 고객한테 보여줬는데. 가타부타 피드백이 없으면 환장하는거유. 고객이 고쳐달라고 했는데 개발자가 고쳐진 결과물을 보여주지 않는다거나. 하는것도 문제가 되지유. 고객이 같이 출근해서 만들어가야 하는 프로젝트이니께. 개발주기는 짧아야 허구. 설계는 단순한 것이 익스트림 프로그래밍을 진행하기에 적당하다 하겠슈.

XP의 주요 실천 방법(Practice)

익스트림 프로그래밍에는 몇가지 실천방법(Practice)가 있는디 다음과 같슈.

  1. Par Programming(짝 프로그래밍)
    • 짝궁하고 같이 개발하는방법이유. 먼일 나면 책임도 같이 지는거쥬.
  2. Collective Onership(공통 코드 소유)
    • 개발 코드에 대한 권한,책임을 공동으로 가지는 것이유.
  3. Test-Driven Development(테스트 주도 개발)
    • 개발자는 코딩전에 테스트 케이스를 먼저 작성해서. 개발 사항을 정확히 파악해야 허유.
    • 테스트가 지속적으로 진행되야 허니깨. 자동화된 테스팅 도구를 사용허유.
  4. Whole Team(전체 팀)
    • 개발에 참여하는 모든 인원(고객 포함)은 각자 역할이 있응께. 책임을 가지고 임하란 말이유.
  5. Continous Intergration(계속적인 통합)
    • 모듈단위로 개발된 코드는 작업 마무리 될 때마다 지속적으로 통합하는 것이유.
  6. Refactoring(리팩토링)
    • 개발된 프로그램의 기능 변경없이 내부코드를 재구성하는 방법이유.
    • 다른 개발자가 코드를 쉽게 이해하고. 유지보수가 용이하게 만들기 위해서 이것을 하는것이유
  7. Small Release(소규모 릴리즈)
    • 수시로 개발된 소프트웨어를 고객한테 보여줘야 허니께. 개발기간은 짧게 텀을 두고 바로바로 고객한테 앱을 보여주는(릴리즈)것이유.

데이터베이스 관리 시스템 (DBMS : Database Management System

데이터베이스 관리 시스템은 사용자와 데이터베이스 사이에 있는 것이유. 사용자가 요청을 하면, 그에 따라 정보를 생성하거나, 조회하거나... 데이터베이스를 관리해주는 소프트웨어를 DBMS 라고 허는 것이유.

DBMS 관련 요구사항 식별시 고려해야 할 사항

  • 가용성
  • 성능
  • 기술지원
  • 상호 호환성
  • 구축 비용

웹 애플리케이션 서버 (WAS : Web Application Server)

사용자의 요구에 따라 변화하는 동적인 콘텐츠를 처리하기 위해 사용하는 미들웨어

요구사항 식별시 고려사항

  • 가용성
  • 성능
  • 기술 지원
  • 구축 비용

오픈소스 (Open Source)

아무나 제한없이 사용할 수 있도록 소스코드가 공개된 소프트웨어

요구사항 식별시 고려사항은 다음이유

  • 라이선스의 종류
  • 사용자 수
  • 기술의 지속 가능성

요구사항

  • 소프트웨어가 제공하는 서비스에 대한 설명
  • 소프트웨어가 정상적으로 운영되는데 핋요한 제약조건

기능 요구사항(Functional Requiremnts)

  1. 시스템이 무었을 하는지 어떤 기능을 하는지
  2. 시스템의 입력이나 출력으로 무었이 포함되어야 하는지
  3. 시스템이 어떤 데이터를 저장하거나 연산을 수행하는지
  4. 시스템이 반드시 수행해야 하는 기능
  5. 사용자가 시스템을 통해 제공받기를 원하는 기능

비기능 요구사항(Non-Functional Requiremnts)

품질이나 제약사항과 관련된 요구사항. 시스템의 장비구성, 성능, 인터페이스, 테스트, 보안 등의 요구사항.

  1. 품질, 제약사항과 관련된 요구사항
  2. 시스템 장비 구성 요구사항
  3. 성능 요구사항
  4. 인터페이스 요구사항
  5. 데이터를 구축하기 위해 필요한 요구사항
  6. 테스트 요구사항
  7. 보안 요구사항
  8. 품질 요구사항 가용성, 정합성, 상호호환성, 대응성, 이식성, 확장성, 보안성 등
  9. 제약사항
  10. 프로젝트 관리 요구사항
  11. 프로젝트 자원 요구사항

요구사항 개발 프로세스

  1. 도출 (Elicitation)
  2. 분석 (Analysis)
  3. 명세 (Specification)
  4. 확인 (Validation)

요구사항 명세

  • 분석된 요구사항을 바탕으로 모델을 작성하고 문서화하는것
  • 기능 요구사항은 빠짐없이
  • 비기능 요구사항은 필요한 것만

요구사항 명세 기법

  • 수학적 원리와 표기법 : 정형 명세기법
  • 자연어 사용 : 비정형 명세기법
구분 정형 명세 기법 비정형 명세 기법
기법 수학적 원리기반, 모델 기반 상태/기능/객체 중심
작성방법 수학적 기호, 정형화된 표기법 일반명사,동사 등 자연어 기반 서술 또는 아이어그램
특징
  • 요구사항을 정확하고 간결하게 표현
  • 요구사항에 대한 결과가 작성자에 상관없이 일관성이 있으므로 완전성 검증 가능
  • 표기업이 어려워 사용자가 이해하기 어려움
  • 자연어 사용 : 요구사항에 대한 결과가 작성자에 따라 다를 수 있음 : 일관성 떨어짐. 해석이 달라질 수 있음
  • 내용 이해가 쉬워 의사소통 용이
종류 VDM, Z, Petri-net, CSP FSM, Decision Table, ER모델링, State Chart(SADT)
